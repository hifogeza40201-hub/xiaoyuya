# 📋 项目管理速查手册

> 适用于敏捷团队的一站式项目管理参考指南

---

## 一、敏捷开发方法论

### 1.1 Scrum 框架

#### 核心角色
| 角色 | 职责 | 时间占比 |
|------|------|----------|
| **Product Owner** | 定义产品愿景、管理Product Backlog、确定优先级 | 全职 |
| **Scrum Master** | 移除障碍、促进流程、保护团队 | 全职 |
| **开发团队** | 自组织完成Sprint目标 | 全职 |

#### 关键仪式
```
📅 Sprint 周期（通常 1-4 周）
├── Sprint Planning    【2-4小时】确定Sprint目标和任务
├── Daily Standup      【15分钟/天】三问：昨天做啥？今天做啥？有障碍吗？
├── Sprint Review      【1-2小时】演示成果，收集反馈
├── Sprint Retrospective【1-2小时】团队反思改进
└── Backlog Refinement 【1-2小时/周】梳理和估算任务
```

#### 工件清单
- **Product Backlog**: 产品级需求清单，按优先级排序
- **Sprint Backlog**: 当前Sprint任务清单
- **Increment**: 可交付的产品增量

#### 关键指标
| 指标 | 计算方式 | 健康阈值 |
|------|----------|----------|
| **Velocity** | 每Sprint完成的Story Points | 波动 < 20% |
| **Sprint目标达成率** | 达成目标Sprint数/总Sprint数 | > 80% |
| **燃尽图斜率** | 剩余工作量/剩余天数 | 接近理想线 |

### 1.2 Kanban 方法

#### 核心原则
1. **可视化工作流程**: 看板展示所有任务状态
2. **限制WIP**: 各阶段设置并行任务上限
3. **管理流动**: 关注任务流转效率
4. **明确流程规则**: 团队共识的定义
5. **持续改进**: 基于数据优化

#### 看板列设计模板
```
📝 To Do → 🔄 In Progress → 👀 In Review → ✅ Done
   [WIP:10]     [WIP:3]         [WIP:2]
```

#### 关键指标
| 指标 | 说明 | 优化目标 |
|------|------|----------|
| **Lead Time** | 需求提出到交付的总时间 | 持续缩短 |
| **Cycle Time** | 开始到完成的时间 | 持续缩短 |
| **Throughput** | 单位时间完成任务数 | 稳定提升 |
| **Flow Efficiency** | 增值时间/总时间 | > 40% |

---

### 1.3 敏捷估算方法

#### Story Points 尺度
| 点数 | 复杂度 | 工作量 | 不确定性 |
|------|--------|--------|----------|
| 1 | 简单 | 少 | 无 |
| 2 | 较简单 | 较少 | 低 |
| 3 | 中等 | 中等 | 中 |
| 5 | 复杂 | 较多 | 较高 |
| 8 | 很复杂 | 多 | 高 |
| 13 | 极复杂 | 非常多 | 极高（需拆分） |

#### 估算技术
- **Planning Poker**: 团队独立估算后讨论差异
- **T-Shirt Sizes**: XS, S, M, L, XL 快速分类
- **Affinity Estimation**: 批量对比分组

## 二、任务管理方法论

### 2.1 任务优先级框架

#### Eisenhower 矩阵
```
                紧急                    不紧急
            ┌─────────────────┬─────────────────┐
            │   🔥 立即做      │   📅 计划做      │
    重要    │  • 线上故障      │  • 技术债        │
            │  • 客户投诉      │  • 能力提升      │
            ├─────────────────┼─────────────────┤
            │   👥 委托他人    │   ❌ 不做        │
   不重要   │  • 常规会议      │  • 低价值需求    │
            │  • 邮件处理      │  • 过度优化      │
            └─────────────────┴─────────────────┘
```

#### MoSCoW 优先级法
- **M**ust have: 必须有，无此功能项目失败
- **S**hould have: 应该有，重要但可妥协
- **C**ould have: 可以有，锦上添花
- **W**on't have: 这次不做，未来考虑

### 2.2 任务分解结构

#### 用户故事模板
```
作为 [角色]，
我想要 [功能/需求]，
以便 [获得价值/目的]。

验收标准：
✓ 条件1（Given-When-Then格式）
✓ 条件2
✓ 条件3
```

#### 任务拆分原则
- **INVEST原则**: Independent, Negotiable, Valuable, Estimable, Small, Testable
- **垂直切片**: 端到端交付，非分层拆分
- **8/13规则**: 超过8点的故事应拆分

### 2.3 时间管理技术

#### Pomodoro 工作法
```
🍅 25分钟专注工作
☕ 5分钟短休息（每4个Pomodoro后15-30分钟长休息）
🚫 专注期间：关闭通知、不切换任务
```

#### Time Blocking
- 将一天划分为深度工作块、会议块、邮件块
- 保护深度工作时间（建议上午2-4小时）

## 三、团队协作规范

### 3.1 会议效率指南

| 会议类型 | 时长限制 | 必备要素 | 取消条件 |
|----------|----------|----------|----------|
| 站会 | 15分钟 | 站立、定时器 | 全员异步更新 |
| 评审会 | 2小时 | 可演示增量 | 无完成内容 |
| 回顾会 | 1小时 | 安全氛围 | 无改进意愿 |
| 计划会 | 4小时 | PO在场、估算 | Backlog未准备 |

### 3.2 沟通渠道规范

```
💬 即时通讯（Slack/钉钉）
   └── 紧急问题、快速确认、非正式讨论

📧 邮件
   └── 正式通知、跨部门沟通、留痕需求

📋 任务系统（Jira/Linear）
   └── 任务追踪、状态更新、技术讨论

📄 文档（Notion/Confluence）
   └── 决策记录、知识沉淀、规范文档

🎥 视频会议
   └── 复杂讨论、头脑风暴、一对一沟通
```

### 3.3 代码与交付规范

#### Git 工作流
```
main (生产分支)
  ↑
develop (开发分支)
  ↑
feature/TICKET-123 (功能分支)
```

#### Definition of Done (DoD)
- [ ] 代码完成并自测
- [ ] 代码审查通过
- [ ] 单元测试覆盖率 > 80%
- [ ] 集成测试通过
- [ ] 文档已更新
- [ ] 产品验收通过
- [ ] 可部署到生产

## 四、风险管理速查

### 4.1 风险识别清单

| 类型 | 常见风险 | 预警信号 |
|------|----------|----------|
| **技术** | 技术债务、第三方依赖故障 | 构建时间增加、Bug率上升 |
| **人员** | 关键人员离职、技能缺口 | 请假增多、任务延期 |
| **需求** | 范围蔓延、需求模糊 | 频繁变更、验收不通过 |
| **进度** | 估算不准、外部依赖 | Velocity下降、阻塞增多 |
| **质量** | 测试不足、架构缺陷 | 线上事故、回归Bug |

### 4.2 风险应对策略
```
规避    → 改变计划消除风险
转移    → 外包、保险、合同条款
减轻    → 增加资源、提前测试、原型验证
接受    → 建立应急储备（时间/预算）
```

## 六、快速参考卡片

```
□ 今日最重要的一件事是什么？
□ 昨日承诺的任务完成了吗？
□ 团队有人被阻塞吗？
□ 今天的会议都有必要吗？

```
